# API — ai-runbook-automation

FastAPI web layer for the autonomous SRE agent. Handles all HTTP endpoints, WebSocket connections, and alert ingestion. Heavy processing is delegated to the `agent-worker` service via Redis/ARQ — the API itself stays lightweight and non-blocking.

---

## Responsibilities

| Concern | Where |
|---------|-------|
| Alert intake from Alertmanager | `POST /alerts/webhook` |
| Alert correlation (dedup within 5 min) | `api/correlation.py` → Redis |
| Job enqueueing | `api/alert_queue.py` → ARQ |
| Incident CRUD reads | `db/incident_store.py` |
| Approval gate signalling | Redis key `approval_decision:{id}` |
| WebSocket broadcast | `api/ws_manager.py` + Redis pub/sub |
| Authentication | `api/auth.py` → `X-API-Key` header |

---

## Request Flow

```
Alertmanager webhook
        │
        ▼
POST /alerts/webhook
        │
        ├─► Alert Correlation (Redis SET NX, 5-min TTL)
        │     duplicate? → silently drop
        │     new?       → continue
        │
        ├─► Create incident row (status=PENDING) in PostgreSQL
        │
        ├─► Enqueue job in ARQ (Redis queue)
        │
        └─► Return WebhookResponse {incidents_queued, incident_ids}

agent-worker processes job:
        │
        ├─► Publishes state changes to Redis channel "incident_updates"
        │
API _redis_listener task:
        │
        ├─► Receives pub/sub messages
        ├─► Persists PENDING_APPROVAL state to PostgreSQL
        └─► Broadcasts raw JSON to all WebSocket clients
```

## Endpoint Reference

### Alerts

#### `POST /alerts/webhook`

Receive an Alertmanager webhook payload. Only `firing` alerts are processed; `resolved` alerts are silently ignored. Duplicate alerts (same service + alertname within the correlation window) are merged into the existing incident.

**Request body** — Alertmanager v4 webhook format:

```json
{
  "version": "4",
  "groupKey": "{}:{alertname=\"HighErrorRate\"}",
  "status": "firing",
  "receiver": "agent-webhook",
  "groupLabels": { "alertname": "HighErrorRate" },
  "commonLabels": { "alertname": "HighErrorRate", "severity": "critical" },
  "commonAnnotations": { "summary": "Error rate above 5%" },
  "alerts": [
    {
      "status": "firing",
      "labels": {
        "alertname": "HighErrorRate",
        "severity": "critical",
        "service": "checkout"
      },
      "annotations": {
        "summary": "Error rate above 5%",
        "description": "Error rate is 8.3% over the last 5 minutes"
      },
      "startsAt": "2024-01-15T10:30:00Z",
      "fingerprint": "abc123"
    }
  ]
}
```

**Response** `200 OK`:

```json
{
  "message": "Queued 1 alert(s) for processing",
  "incidents_queued": 1,
  "incident_ids": ["inc_abc123"]
}
```

---

#### `POST /simulate`

Same as `/alerts/webhook` but forces `APPROVAL_MODE=DRY_RUN`. Use this for testing without executing real remediation actions. Accepts the same Alertmanager payload format.

### Incidents

#### `GET /incidents`

List all incidents, ordered by most recent. Returns lightweight summaries — no transcript or full analysis.

**Response** `200 OK` — array of `IncidentSummary`:

```json
[
  {
    "incident_id": "inc_abc123",
    "alert_name": "HighErrorRate",
    "status": "RESOLVED",
    "summary": "Restarted checkout service; error rate dropped to 0.1%",
    "actions_taken_count": 3,
    "started_at": "2024-01-15T10:30:00Z",
    "resolved_at": "2024-01-15T10:34:22Z",
    "duration_seconds": 262.0
  }
]
```

---

#### `GET /incidents/{incident_id}`

Full incident detail. Includes the complete LLM reasoning transcript, all action records, and the generated PIR (if resolved).

**Path parameter:** `incident_id` — UUID string (e.g. `inc_abc123`)

**Response** `200 OK` — `Incident` object:

```json
{
  "incident_id": "inc_abc123",
  "alert_name": "HighErrorRate",
  "alert": { "labels": { ... }, "annotations": { ... } },
  "status": "RESOLVED",
  "summary": "Restarted checkout service; error rate dropped to 0.1%",
  "root_cause": "Memory leak in connection pool causing elevated error rate",
  "actions_taken": [
    {
      "action": "get_metrics",
      "params": { "query": "rate(http_requests_total{status=~'5..'}[5m])" },
      "result": "SUCCESS",
      "output": "0.083",
      "duration_ms": 145,
      "timestamp": "2024-01-15T10:30:12Z"
    }
  ],
  "recommendations": ["Increase connection pool size", "Add OOM alerting"],
  "reasoning_transcript": [ ... ],
  "state_history": [ ... ],
  "started_at": "2024-01-15T10:30:00Z",
  "resolved_at": "2024-01-15T10:34:22Z",
  "pir": { ... }
}
```

**Error** `404` if incident not found.

---

#### `GET /incidents/{incident_id}/pir`

Returns the auto-generated Post-Incident Review for a resolved incident. The PIR is generated by the LLM after resolution and contains severity, root cause, timeline, action items, and prevention checklist.

**Error** `404` if incident not found or PIR not yet generated (still processing).

---

#### `POST /incidents/{incident_id}/approve`

Approve a pending destructive action. Unblocks the waiting `ApprovalGate` in the worker process via a Redis key `approval_decision:{id}=APPROVED`.

**Request body** (all fields optional):

```json
{
  "action": "restart_service",
  "operator": "oncall-engineer",
  "reason": "Confirmed memory leak — safe to restart"
}
```

**Response** `200 OK` — `ApprovalResponse`:

```json
{
  "incident_id": "inc_abc123",
  "action": "restart_service",
  "approved": true,
  "operator": "oncall-engineer",
  "reason": "Confirmed memory leak — safe to restart",
  "responded_at": "2024-01-15T10:33:01Z"
}
```

**Error** `404` if incident not found.

---

#### `POST /incidents/{incident_id}/reject`

Reject the pending action. The agent skips the action and continues its reasoning loop.

**Request body** (all fields optional):

```json
{
  "action": "restart_service",
  "reason": "Service is stateful — restart would cause data loss"
}
```

**Response** `200 OK` — same `ApprovalResponse` shape with `"approved": false`.

### Runbooks

#### `GET /runbooks`

List all auto-discovered YAML runbooks from the `runbooks/` directory.

**Response** `200 OK` — array of runbook objects:

```json
[
  {
    "name": "high_error_rate",
    "description": "Handles HighErrorRate alerts ...",
    "triggers": ["HighErrorRate"],
    "action_count": 5,
    "actions": ["get_metrics: ...", "get_recent_logs: ...", "restart_service: ..."],
    "escalation_threshold": "Escalate if error rate remains above 5% after restart",
    "metadata": { "severity": "P2", "team": "platform", "runbook_version": "1.0" }
  }
]
```

---

#### `GET /runbooks/{name}`

Single runbook by name (e.g. `GET /runbooks/high_error_rate`). Same shape as a list item.

**Error** `404` if runbook not found.

---

#### `GET /runbooks/{name}/yaml`

Raw YAML file content as `text/plain`. Used by the UI inline editor to populate the textarea.

**Error** `404` if runbook not found.

---

#### `POST /runbooks`

Create a new runbook from raw YAML.

**Request body:**

```json
{ "yaml": "name: my_runbook\ntriggers:\n  - MyAlert\n..." }
```

The `name` field inside the YAML determines the filename. Returns `409 Conflict` if a runbook with that name already exists.

**Response** `200 OK`:

```json
{ "created": "my_runbook", "path": "my_runbook.yml" }
```

**Errors:** `409` duplicate, `422` invalid YAML or missing `name`.

---

#### `PUT /runbooks/{name}`

Update an existing runbook. The `name` field inside the YAML must match the URL parameter.

**Request body:** same as `POST /runbooks`.

**Response** `200 OK`:

```json
{ "updated": "my_runbook", "path": "my_runbook.yml" }
```

**Errors:** `422` if YAML `name` doesn't match URL, `422` invalid YAML.

### Agent Mode

#### `GET /agent/mode`

Returns the current agent approval mode, reading from Redis first (falls back to the `APPROVAL_MODE` env var if Redis is unavailable).

**Response** `200 OK`:

```json
{ "mode": "AUTO" }
```

Possible values: `AUTO`, `MANUAL`, `DRY_RUN`.

---

#### `POST /agent/mode`

Change the agent approval mode at runtime. Takes effect for the next incident processed.

**Request body:**

```json
{ "mode": "MANUAL" }
```

**Error** `422` if mode is not one of the valid values. `503` if Redis is unreachable.

### Observability

#### `GET /health`

Checks reachability of all external dependencies. Safe to call without authentication — bypasses `API_KEY` check.

**Response** `200 OK` — `HealthResponse`:

```json
{
  "status": "healthy",
  "claude_api": "reachable",
  "prometheus": "reachable",
  "redis": "reachable",
  "queue_depth": 0,
  "active_workers": 0,
  "incidents_processed": 0
}
```

`status` is `"healthy"` when LLM and Redis are reachable, `"degraded"` otherwise. The `claude_api` field reflects whichever LLM backend is configured (`LLM_BACKEND`).

#### `GET /correlations`

Active alert correlation groups in memory. Useful for debugging alert storms.

**Response** `200 OK`:

```json
{
  "active_groups": [
    { "key": "checkout:HighErrorRate", "incident_id": "inc_abc123", "expires_in_seconds": 180 }
```

---

#### `GET /metrics`

Prometheus scrape endpoint. Exposes agent metrics: MTTR histogram, action counts, token usage, and incident counters. Bypasses authentication.

---

#### `POST /simulator/scenario`

Switch the active scenario served by mock-prometheus at runtime. For demo and local testing only.

**Request body:**

```json
{ "scenario": "high_error_rate" }
```

**Error** `400` if `scenario` is missing. `502` if mock-prometheus is unreachable.

### WebSocket — `ws://host/ws`

Opens a persistent connection for real-time incident state updates. The API bridges Redis pub/sub channel `incident_updates` to all connected WebSocket clients.

**Connection URL:**

```
ws://localhost:8000/ws
ws://localhost:8000/ws?api_key=<value>   # when API_KEY is set
```

**Incoming message format** (JSON):

All messages share an `incident_id` field. The `status` field mirrors `IncidentStatus`.

```json
{ "incident_id": "inc_abc123", "status": "PROCESSING" }
```

```json
{
  "incident_id": "inc_abc123",
  "status": "PENDING_APPROVAL",
  "pending_action": "restart_service",
  "params": { "service": "checkout" },
  "sre_insight": {
    "reason": "Memory usage at 94% — restart will reset the connection pool",
    "params": { "service": "checkout" }
  }
}
```

```json
{ "incident_id": "inc_abc123", "status": "RESOLVED" }
```

**Known status values pushed over WebSocket:**

| Status | Meaning |
|--------|---------|
| `PENDING` | Incident created, queued |
| `PROCESSING` | Agent actively working |
| `PENDING_APPROVAL` | Waiting for human approval on a destructive action |
| `RESOLVED` | Agent resolved the incident |
| `ESCALATED` | Agent escalated to Slack/PagerDuty |
| `FAILED` | Agent loop failed (see worker logs) |
| `OBSERVING` / `REASONING` / `ACTING` / `VERIFYING` | Sub-states during active processing |

**Keep-alive:** Send any text frame to keep the connection open. The server does not send pings.

## Authentication

When `API_KEY` is set in the environment, every request must include:

```
X-API-Key: <value>
```

Alternatively, pass as a query parameter:

```
GET /incidents?api_key=<value>
```

**Exempted paths** (no auth required): `/health`, `/metrics`, `/docs`, `/openapi.json`, `/redoc`

If `API_KEY` is unset, authentication is disabled (development mode).

**Error** `401 Unauthorized` with `WWW-Authenticate: ApiKey` header when the key is missing or wrong.

## Error Envelope

All HTTP errors from FastAPI follow this shape:

```json
{ "detail": "Incident 'inc_xyz' not found" }
```

For validation errors (`422 Unprocessable Entity`):

```json
{
  "detail": [
    {
      "loc": ["body", "yaml"],
      "msg": "field required",
      "type": "value_error.missing"
    }
  ]
}
```

## Pydantic Models

Located in `api/models.py`. Key types used across request/response:

| Model | Used for |
|-------|---------|
| `AlertmanagerWebhook` | Webhook + simulate request body |
| `Alert` | Single alert within a webhook payload |
| `AlertLabels` | Alert label set (extra fields allowed) |
| `AlertAnnotations` | Alert annotation set (extra fields allowed) |
| `IncidentSummary` | `GET /incidents` list items |
| `Incident` | `GET /incidents/{id}` full detail |
| `ActionRecord` | Individual tool call within an incident |
| `ReasoningStep` | Single turn in the LLM transcript |
| `ApprovalResponse` | Approve/reject response |
| `WebhookResponse` | Webhook/simulate response |
| `HealthResponse` | `/health` response |

**`IncidentStatus` enum values:**

`PENDING` · `PROCESSING` · `RESOLVED` · `ESCALATED` · `FAILED` · `PENDING_APPROVAL` · `DETECTED` · `OBSERVING` · `REASONING` · `ACTING` · `VERIFYING`

## Environment Variables

Variables consumed directly by the API service:

| Variable | Default | Description |
|----------|---------|-------------|
| `DATABASE_URL` | `postgresql+asyncpg://sre:sre@postgres:5432/runbooks` | Async PostgreSQL connection string |
| `REDIS_URL` | `redis://redis:6379` | Redis (queue + pub/sub + approval signalling) |
| `PROMETHEUS_URL` | `http://mock-prometheus:9091` | Used by `/health` reachability check |
| `API_KEY` | *(unset)* | Optional API key; disables auth if empty |
| `APPROVAL_MODE` | `AUTO` | Default approval mode (`AUTO` \| `DRY_RUN` \| `MANUAL`) |
| `CORRELATION_WINDOW_SECONDS` | `300` | Alert dedup window in seconds |
| `LLM_BACKEND` | `ollama` | Which LLM backend to health-check (`ollama` \| `claude`) |
| `OLLAMA_URL` | `http://host.docker.internal:11434` | Ollama URL (for health check only) |
| `ANTHROPIC_API_KEY` | *(unset)* | Used by health check when `LLM_BACKEND=claude` |
| `CORS_ORIGINS` | `http://localhost:3000` | Comma-separated allowed CORS origins |
| `LOG_LEVEL` | `INFO` | Python logging level |

## Local Development

### With Docker Compose (recommended)

```bash
# From project root
cp .env.example .env
docker compose up -d api
```

The API will be available at `http://localhost:8000`. OpenAPI docs at `http://localhost:8000/docs`.

### Without Docker

```bash
# Install dependencies
pip install -r requirements.txt

# Set required env vars
export DATABASE_URL="postgresql+asyncpg://sre:sre@localhost:5432/runbooks"
export REDIS_URL="redis://localhost:6379"

# Start
uvicorn api.main:app --reload --port 8000
```

### Interactive API Docs

FastAPI auto-generates documentation served at:

- `http://localhost:8000/docs` — Swagger UI (try endpoints directly)
- `http://localhost:8000/redoc` — ReDoc (read-only, cleaner format)
- `http://localhost:8000/openapi.json` — Raw OpenAPI schema

### Running API Tests

```bash
# No real services needed — everything is mocked
pytest tests/test_api.py -v

# Full suite
pytest tests/ -v
```
